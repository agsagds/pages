Общие подходы к AI-powered development

Ask → Plan → Build

Агент может сразу кодить и решить задачу мимо цели.
Поэтому: отдельно исследование / план / реализация.
В доке 4 фазы (Explore → Plan → Implement → Commit); в докладе — Ask → Plan → Build.

Источник: Claude Code best practices

Ask / Explore

Plan mode: чтение кода, ответы на вопросы, без правок.

Plan

Файлы, flow, edge cases. План можно править до старта.

Build

Реализация по плану + проверка pass / fail.

План нужен

Неясный подход · много файлов · незнакомый код

Сразу Build

Diff в одно предложение: typo, log, rename

Закрой loop

Без check «выглядит готово» — verification loop остаётся на человеке.
Check: tests · build exit code · linter · screenshot vs design.
Нужен evidence: лог, вывод команды.

Ask-Plan-Build без verify = квадратные колёса.

Контекст решает!

Контекст = переписка + прочитанные файлы + вывод команд.

Заполняется быстро → качество падает («забывает» инструкции).
Им управляют сознательно — как бюджетом.

Системные промпты и дефолтный набор инструментов формируют заметную разницу в качестве.

Как обеспечить наблюдаемость в карточке?

something better than nothing

Минимальный стандарт прозрачности в карточке полезнее, чем отсутствие структуры.

Цель задачи формулируется в одном абзаце: что меняется и какой ожидается эффект.
Ход работы фиксируется короткими этапами: исследование, реализация, проверка.
Статус обновляется через наблюдаемые факты: ссылка на diff, номер билда, выдержка из логов.
Результат закрывается критерием готовности: какие проверки пройдены и что осталось за рамками.

Минимальный шаблон

Контекст → План → Изменения → Проверка → Риски

Частота обновлений

После каждого завершённого шага и перед сменой направления.

забери свой нейрослоп!

Текст от модели превращается в рабочий материал только после фильтрации и верификации.

Плотные абзацы разбиваются на шаги, которые можно проверить отдельно.
Каждый технический тезис подтверждается ссылкой на код, документ или воспроизводимую команду.
Любое «звучит убедительно» проверяется на уровне runtime-сигналов и edge cases.
Неоднозначные места помечаются открытыми вопросами, а не скрываются под общими формулировками.
Фильтр качества

Факт · источник · проверка · влияние на продукт.

Холивар

затраты на «держать лицо»

Попытка скрыть неопределённость обычно дороже, чем ранняя фиксация риска.

Скрытый риск

Проблема обнаруживается поздно, когда стоимость изменения выше.

Открытый риск

Риск фиксируется сразу: гипотеза, план проверки и срок решения.

Нормальная практика: «не подтверждено», «требуется проверка», «нужны данные».
В карточке хранится не образ уверенности, а траектория принятия решений.

инженер — не кодер

Ценность создаётся не количеством строк, а устойчивостью решения и скоростью обратной связи.

Инженерная роль: уточнение задачи, работа с ограничениями, оценка влияния.
Код — один из артефактов наряду с тестами, наблюдаемостью и документацией решения.
AI-инструменты ускоряют реализацию, но ответственность за архитектуру и риски остаётся в команде.

Операционный критерий

Решение считается готовым, если его можно поддерживать без автора через месяц.

AI-first лемминги

AI-first работает как стратегия только при явных границах применения.

Где уместно

Быстрые прототипы, рутинные миграции, генерация тестовых заготовок.

Где нужен ручной контроль

Критичные доменные правила, безопасность, платежные и правовые сценарии.

Внедрение начинается с пилота и ограниченного контура ответственности.
Метрики внедрения: lead time, дефекты после релиза, доля ручных откатов.
Решения по расширению принимаются по данным, а не по тренду.
